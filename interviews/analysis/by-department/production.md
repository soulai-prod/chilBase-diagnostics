## Секция 1: Кодированный транскрипт

### 1.1 [ROUTINE] — Регулярные процессы и воркфлоу

**R-01. Полный цикл разработки фичи**  
8-этапный процесс: vision doc → обсуждение с продюсерами → передача в разработку → вычитка GD-документации → декомпозиция → разработка → тестирование → релиз. Повторяющийся конвейер для каждой фичи.

**R-02. Автоматическое движение задач по статусам**  
Уже в разработке — автоматизация перехода задач между статусами в таск-трекере. Классическая процессная автоматизация (не AI).

**R-03. Уведомления о коммитах в чат**  
Автоматические нотификации при коммитах — интеграция GitLab/VCS с мессенджером.

**R-04. Массовые операции с доступами**  
Пример Андрея: операции с 50 пользователями одновременно. Рутинная административная задача, повторяющаяся при онбординге/оффбординге.

**R-05. Code review**  
[ДАННЫЕ НЕДОСТАТОЧНЫ] Процесс code review упомянут в контексте неудачной автоматизации. Частота, формат и трудозатраты на ручной review не зафиксированы в транскрипте.

### 1.2 [PAIN] — Явные боли

**P-01. Неудачная автоматизация code review**  
Попробовали несколько готовых решений в GitLab — качество неприемлемое. Нейросеть «писала стихи» вместо полезных комментариев. Результат: скептицизм к AI-решениям для кода.

**P-02. Нехватка времени на настройку AI-инструментов**  
Все заняты production-задачами, свободного слота на эксперименты нет. Паттерн, идентичный маркетингу (P-05 в Marketing).

**P-03. Вопросы безопасности корпоративного кода**  
Запрет на передачу кода внешним AI-сервисам. Создаёт конфликт: хочется использовать мощные внешние модели (ChatGPT, Claude), но нельзя отправлять код.

**P-04. AI-помощник по кодовой базе ненадёжен**  
Может «наврать» о зависимостях. Требует контекста всех репозиториев (клиент + сервер + инфраструктура). Подходит только для простых запросов типа «где найти метод».

### 1.3 [SCALE_RISK] — Что ломается при масштабировании

**SR-01. Code review при росте команды**  
[ДАННЫЕ НЕДОСТАТОЧНЫ] При x2 разработчиков нагрузка на code review растёт нелинейно. Попытка автоматизировать уже провалилась. Остаётся ручной процесс — потенциальное бутылочное горлышко.

**SR-02. Декомпозиция и вычитка GD-документации**  
[ДАННЫЕ НЕДОСТАТОЧНЫ] При росте числа фичей — больше vision docs, больше обсуждений с продюсерами, больше вычиток. Масштабирование не обсуждалось в транскрипте, но архитектурно это узкое место.

**SR-03. Массовые операции с доступами**  
При росте компании x2 (200 → 400 чел.) — объём рутинных операций с доступами масштабируется линейно. Андрей уже упомянул кейс с 50 пользователями — при x2 это 100+.

### 1.4 [DATA_ASSET] — Источники данных

**DA-01. Кодовая база (GitLab)**  
Клиентская, серверная и инфраструктурная часть. Потенциальный датасет для AI code assistant, но с ограничениями безопасности.

**DA-02. История коммитов и merge requests**  
Данные о паттернах разработки, частоте ошибок, времени на review — потенциальный датасет для предиктивной аналитики.

**DA-03. GD-документация (vision docs)**  
Структурированные документы о фичах — могут использоваться для RAG-системы или AI-ассистента по продукту.

[ДАННЫЕ НЕДОСТАТОЧНЫ] Нет информации о: системах мониторинга, логах, CI/CD метриках, баг-трекинге — всё это потенциальные data assets для AI.

### 1.5 [AI_SIGNAL] — Текущее использование AI, отношение, инструменты

**AI-01. Массовое использование AI для генерации кода**  
«Многие разработчики» используют AI. Детали (какие инструменты, для каких задач, частота) — не зафиксированы.

**AI-02. CTO использует AI для массовых рефакторингов**  
Значимый сигнал: руководство технического блока лично использует AI для продуктивных задач. Сверху-вниз adoption.

**AI-03. Внутренний сервер с локальными моделями (!)**  
Критически важный сигнал: компания уже имеет инфраструктуру для запуска AI-моделей внутри контура. Решает проблему безопасности кода. Детали (какие модели, мощность, кто использует) — не зафиксированы.

**AI-04. ChatGPT у Ильи и Андрея**  
Оба респондента используют ChatGPT для различных задач. Конкретные кейсы не описаны.

**AI-05. Скептицизм после провала code review**  
Негативный AI-сигнал. Неудачный опыт формирует настороженность. «Нейросеть писала стихи» — яркий анти-пример, который может распространяться как нарратив в команде.

**AI-06. Обучение через YouTube, статьи, peer learning**  
Неформальное обучение. Особо выделяется обучение от «Саши» (вероятно, Саша Ломатов из Performance Marketing — тот же AI Champion, которого мы идентифицировали ранее). Живые демо от лидов команд.

### 1.6 [DEPENDENCY] — Критические зависимости

**D-01. Зависимость от внутреннего сервера с локальными моделями**  
Кто его поддерживает? Какие модели? Какая мощность? Если сервер упадёт — теряется ключевая возможность безопасного использования AI. [ДАННЫЕ НЕДОСТАТОЧНЫ для полной оценки.]

**D-02. Зависимость от GD-команды для вычитки документации**  
Каждая фича проходит через вычитку с GD — последовательная зависимость в pipeline.

**D-03. Зависимость от CTO как AI-драйвера**  
CTO применяет AI для рефакторингов — потенциально единичная точка отказа для технического AI-adoption.

---

### 1.7 [HIDDEN_PAIN] — Скрытые боли

**HP-01. Разрыв между «AI для кода» и «AI для процессов»**  
Разработчики используют AI для генерации кода, но процессы (vision doc → декомпозиция → review) остаются полностью ручными. Автоматизация смотрит только на код, не на workflow вокруг кода.

**HP-02. Скептицизм как self-fulfilling prophecy**  
Неудача с code review → скептицизм → меньше попыток → меньше успехов → больше скептицизма. Цикл может блокировать adoption даже полезных решений.

**HP-03. Безопасность как тормоз adoption**  
Запрет на внешние сервисы логичен, но локальные модели (внутренний сервер) — компромиссное решение с ограниченными возможностями. Команда может не получать преимущества от передовых моделей (Claude, GPT-4), оставаясь на менее мощных локальных.

**HP-04. Отсутствие формализованного knowledge sharing по AI**  
Обучение через YouTube и «от Саши» — неструктурированно. При 36 разработчиках масштаб peer learning ограничен. Нет программы, нет стандарта.

---

### 1.8 [CROSS_DEPT] — Кросс-департаментные зависимости

**CD-01. Саша Ломатов как мост между Marketing и Production**  
Саша из Performance Marketing обучает разработчиков AI-навыкам. Кросс-департаментный knowledge transfer, не формализованный.

**CD-02. Внутренний сервер с локальными моделями → вся компания**  
Если сервер обслуживает не только Production, это общекорпоративная инфраструктура. Кто управляет? Какие SLA? Может ли маркетинг использовать?

**CD-03. GD-команда ↔ Production**  
Vision docs и вычитка документации — точка пересечения Product и Engineering. Потенциальная зона для AI-ассистента.

**CD-04. Процессы продюсирования (Андрей) → все продуктовые отделы**  
Head of Production координирует между отделами. Его видение AI-возможностей для процессов влияет на все команды.

### 1.9 Пробелы в данных

Из-за сокращённого формата транскрипта отсутствует информация по:

| Тема | Что не хватает | Влияние на анализ |
|------|---------------|-------------------|
| Конкретные AI-инструменты разработчиков | Copilot? Cursor? Cody? Какие именно | Невозможно оценить tech stack для AI |
| Детали внутреннего сервера | Модели, мощность, usage stats | Невозможно оценить зрелость AI-инфраструктуры |
| Метрики процессов | Время цикла фичи, время на review, backlog size | Невозможно количественно оценить impact инициатив |
| CI/CD и DevOps | Уровень автоматизации pipeline | Пропущен целый слой потенциальных AI-применений |
| Тестирование (QA взаимодействие) | Как Production взаимодействует с QA | Кросс-департаментная зависимость не оценена |
| Барьеры adoption в деталях | Конкретные истории сопротивления | Невозможно проектировать change management |
| Данные о мониторинге и observability | Системы логирования, алертинга | Потенциальный слой AI-применений не виден |

## Секция 2: Оценка AI-зрелости по Deloitte (7 измерений)

### 2.1 Стратегия и roadmap — **1.5 / 5.0**

**Обоснование:** Нет упоминания AI-стратегии на уровне отдела. AI используется ad hoc: кто хочет — генерирует код, CTO рефакторит. Попытка автоматизировать code review была реактивной, не частью плана. Нет roadmap, нет KPI. Даже автоматизация задач (движение статусов) — это не AI, а классическая автоматизация.

**Данные достаточны:** Частично. Отсутствие стратегии хорошо видно из контекста, но нельзя исключить, что стратегия существует на уровне CTO и не была озвучена.

### 2.2 Таланты и культура — **2.5 / 5.0**

**Обоснование:** Сильная сторона — команда сеньоров, технически способная осваивать AI. CTO лично использует AI. Многие разработчики генерируют код с AI. Peer learning от Саши и лидов. Слабая сторона: скептицизм после провала code review, нехватка времени, нет формального обучения, нет AI-культуры как нормы. Знания распределены неравномерно.

**Данные достаточны:** Частично. Нет данных о распределении навыков внутри 36 человек.

### 2.3 Данные — **2.0 / 5.0**

**Обоснование:** Кодовая база и GD-документация существуют, но не подготовлены для AI-использования. AI-помощник по кодовой базе признан ненадёжным — «может наврать о зависимостях», нужен контекст всех репозиториев. Это означает: данные есть, но не структурированы, не индексированы, не готовы для AI.

**Данные достаточны:** Слабо. Нет информации о метриках разработки, мониторинге, качестве данных.

### 2.4 Технологии и платформы — **3.0 / 5.0**

**Обоснование:** Ключевой дифференциатор: **внутренний сервер с локальными моделями**. Это единственный отдел (из проанализированных), который имеет собственную AI-инфраструктуру. Плюс: разработчики массово используют AI для кодогенерации, CTO применяет для рефакторингов. GitLab-интеграции пробовались. Минус: попытка code review провалилась, AI codebase assistant ненадёжен, нет единой платформы.

**Данные достаточны:** Частично. Нет деталей о внутреннем сервере (модели, мощность, usage).

### 2.5 Governance и этика — **2.0 / 5.0**

**Обоснование:** Есть одно чёткое правило: запрет на передачу корпоративного кода внешним сервисам. Это базовый governance. Но: нет формальной AI-политики для разработчиков, нет гайдлайнов по использованию AI-сгенерированного кода, нет процесса проверки AI-output в code review. Внутренний сервер — техническое решение проблемы безопасности, но не governance-framework.

**Данные достаточны:** Слабо. Возможно, существует больше правил, которые не были озвучены.

### 2.6 Экосистема и партнёры — **1.5 / 5.0**

**Обоснование:** Нет упоминания AI-партнёрств, вендоров, внешних интеграций (за пределами попыток с GitLab-решениями). Внутренний сервер — in-house решение. Обучение через YouTube и статьи, не через партнёров или программы обучения.

**Данные достаточны:** Слабо. Транскрипт мог не покрыть эту тему.

### 2.7 Измерение ценности — **1.0 / 5.0**

**Обоснование:** Нет ни одного упоминания метрик эффективности AI. Даже провал code review описан качественно («стихи писала»), а не количественно (сколько ложных срабатываний, какой % полезных комментариев). Нет baseline-метрик, нет before/after.

**Данные достаточны:** Достаточны для низкой оценки. Отсутствие метрик — само по себе данные.

**Парадокс:** Самый технически продвинутый отдел (сеньоры, внутренний сервер, AI-кодогенерация) — но зрелость ниже маркетинга. Причина: техническая инфраструктура без стратегии, governance и измерения ценности не даёт прироста зрелости.

### 5.2 Парадокс технической зрелости

**Ключевой инсайт:** Production — самый технически продвинутый отдел из проанализированных (сеньоры, внутренний сервер с локальными моделями, массовое AI-использование для кода). Но AI-зрелость *ниже* маркетинга. Причины:

1. **Техническая инфраструктура без стратегии = assets, не capabilities.** Сервер есть, но нет плана, как его использовать. Разработчики генерируют код, но нет стандартов, метрик, процессов.

2. **Провал code review создал негативный якорь.** В маркетинге Саша двигает позитивный нарратив («Claude Code + ClickHouse работает!»). В Production нарратив негативный («нейросеть писала стихи»). Это влияет на культуру и готовность к экспериментам.

3. **«Инженерный скептицизм» vs. «маркетинговый прагматизм».** Инженеры требуют от AI точности (код должен работать, зависимости должны быть верными). Маркетологи принимают probabilistic output (креатив может быть неидеальным, итерируем). Разные стандарты качества → разные пороги adoption.

### 5.3 Паттерн: «Тонем в операционке» (подтверждение)

Второй отдел подряд называет нехватку времени как барьер. В маркетинге: «даже при желании нет свободной недели». В Production: «все заняты production». Это **системная, компанейская проблема**, а не особенность отдела.

**Рекомендация (усилена):** Формальное выделение AI-времени (20% time, AI-день, или хотя бы 4 часа в неделю) — must have для любого AI-roadmap. Без этого все инициативы останутся на бумаге.

### 5.4 Паттерн: Peer learning через Сашу Ломатова (кросс-департаментный)

Саша из Performance Marketing обучает инженеров AI-навыкам. Это подтверждает его статус AI Champion и показывает: **знания распространяются горизонтально, не вертикально**. Руководство не создаёт программы обучения — люди учатся друг у друга.

Это одновременно сила (органический adoption) и слабость (bus factor = 1, нет масштабирования, зависимость от одного человека).

### 5.5 Новый паттерн: Внутренний AI-сервер как стратегический актив

Ни в одном другом проанализированном отделе нет упоминания **собственной AI-инфраструктуры**. Это уникальное преимущество Production:

- Решает проблему безопасности кода (главный барьер отдела)
- Потенциально решает проблему AI blacklist для всей компании (если модели разрешены внутри контура)
- Может стать **корпоративной AI-платформой**, а не только инструментом Production

**Критический вопрос (для выяснения):** Кто управляет этим сервером? Какие модели? Какая мощность? Доступен ли другим отделам? Эта информация может изменить весь AI-roadmap компании.

### 5.6 Новый паттерн: CTO как скрытый AI-драйвер

CTO использует AI для массовых рефакторингов. Это не было зафиксировано в маркетинговом интервью. CTO — потенциально самый влиятельный AI-амбассадор в компании: техническая экспертиза + authority + visibility.

**Рекомендация:** Обязательно включить CTO в программу амбассадоров. Если CTO станет формальным спонсором AI-инициатив — это ускорит adoption во всех технических отделах.

### 5.7 Паттерн: Отсутствие измерения ценности (подтверждение)

Второй отдел подряд: никто не считает ROI от AI. В маркетинге — нет метрик по AI creative pipeline. В Production — нет метрик по AI-кодогенерации. Без метрик невозможно: (1) обосновать инвестиции, (2) выбрать приоритеты, (3) показать результат руководству.

### 5.8 Инсайт: Разная природа AI-скептицизма

| | Marketing | Production |
|---|-----------|------------|
| **Источник скептицизма** | Страх замены, сложность, недоверие | Неудачный опыт (code review), ненадёжность (codebase assistant) |
| **Тип** | Эмоциональный | Рациональный |
| **Стратегия преодоления** | Показать кейсы, дать попробовать | Показать *качественное* решение, дать контроль, начать с малого |

Production-скептицизм *рациональнее*: они попробовали, не сработало. Это сложнее преодолеть, чем эмоциональное сопротивление. Но зато, когда решение *работает* — engineers примут его быстрее.

### 5.9 Рекомендация по подходу к Production

1. **Не начинать с code review** (травма). Начать с обучения и простых инструментов.
2. **Использовать внутренний сервер** как безопасную площадку для экспериментов.
3. **Привлечь CTO** как спонсора и демонстратора.
4. **Метрики с первого дня:** замерить время до/после для каждого AI-эксперимента.
5. **Дать инженерам контроль:** не «вот вам AI-инструмент», а «вот платформа, настройте под себя».
